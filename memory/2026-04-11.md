### 핵심 발견: Gemma vs OpenAI tool calling 포맷 불일치

붐엘 서버의 tool calling 구현 (server_v3_postgres_router.py, line 1389~1476):

**Gemma 4 고유 포맷 (서버가 파싱하는 형식):**
```
<|tool_call>call:함수명{인자1:<|"|>값<|"|>}<tool_call|>
```

**OpenAI/OpenClaw 포맷 (외부에서 기대하는 형식):**
```json
{"tool_calls": [{"id": "call_xxx", "type": "function", "function": {"name": "...", "arguments": "..."}}]}
```

**결론:** 붐엘 서버는 Gemma의 tool call 출력을 **서버 내부에서 완전히 소화**한 후 최종 텍스트만 반환.
OpenClaw에게 "tool 써야 해, 실행해줘" 신호(OpenAI tool_calls 포맷)를 **전혀 보내지 않음**.

→ OpenClaw가 에이전틱 루프를 관리하려면 **포맷 변환 레이어가 필요**.

---

### 에이전틱 통합을 위한 필요 작업 (설계)

OpenClaw가 붐엘의 에이전틱 루프를 관리하려면:

1. **입력 변환**: OpenClaw가 보낸 OpenAI tools → Gemma tool 포맷으로 변환 후 모델 전달
2. **출력 변환**: Gemma의 `<|tool_call>call:xxx{...}<tool_call|>` → OpenAI `tool_calls` 포맷으로 변환 후 반환 (서버 내부 실행 안 함)
3. **결과 주입**: OpenClaw가 tool 실행 후 결과를 다시 보내면 → Gemma `<|tool_response|>` 포맷으로 변환

이 레이어가 없으면:
- A안(`supportsTools: false`): wiki/memory 유지, OpenClaw 도구 미사용, 에이전틱 개선 없음
- B안(`supportsTools: true`): OpenClaw tools가 Gemma에 전달되나 Gemma가 무시하고 환각 → 더 나빠질 수 있음

### 디버깅 과정 상세

#### 문제 1: content: [] (빈 응답)

**증상**: OpenClaw가 붐엘에 요청하고 200 OK를 받지만 `content: []`, `usage: {input: 0, output: 0}`

**원인 분석 과정**:
1. `booml_raw_server.py`에 `[DEBUG] response_text repr: ...` 추가 → 서버는 정상 텍스트 생성 확인

#### 문제 2: 60초 타임아웃 초과

**증상**: 스트리밍 구현 후에도 타임아웃 오류 `"Request timed out before a response was generated"`

**원인**:

**참고**: 현재 SSE 구현은 전체 생성 완료 후 청크 전송 (가짜 스트리밍). 진짜 토큰 단위 스트리밍은 미구현.

### 특이점 및 주의사항 (업데이트)

1. **`openclaw nodes approve` vs `openclaw devices approve`** ← 핵심 주의
   - OpenClaw에는 두 개의 완전히 별도인 pairing 시스템이 존재
   - `nodes/pending.json` → node-host 연결 → `openclaw nodes approve <requestId>`
   - `devices/pending.json` → CLI 디바이스 역할 → `openclaw devices approve <requestId>`
   - gateway 재시작 후 node 루프 빠지면 반드시 `nodes/` 확인 먼저

2. **세션 캐시 MiniMax 오버라이드**
   - 파일: `/Users/sykim/.openclaw/agents/booml/sessions/sessions.json`
   - `agent:booml:telegram:direct:47980209` 엔트리에 `modelOverride`, `providerOverride` 필드가 생기면 이게 config보다 우선함
   - MiniMax가 응답하면 해당 세션 엔트리 삭제 후 gateway NOT restart, node restart만

3. **MLX 컨텍스트 길이 문제**
   - OpenClaw 시스템 프롬프트 + 도구 스키마 = 약 19,000~20,000 토큰
   - 프리필만 39초 소요 (20k 토큰 × ~520 tok/s)
   - 대화가 길어질수록 더 느려짐 → 미래 과제: workspace 파일 최소화

4. **stream=True 가짜 스트리밍**
   - 현재 구현은 전체 생성 완료 후 청크 분할 전송
   - 진짜 스트리밍 미구현 → 첫 토큰 응답 지연 없음 (프리필 39초 후 한 번에 수신)
   - 실질적 영향: OpenClaw 타임아웃 카운트가 전체 생성 시간 기준으로 작동

5. **openclaw.json booml 에이전트 두 세션 키**
   - `agent:booml:main` — CLI/내부 세션
   - `agent:booml:telegram:direct:47980209` — 텔레그램 봇 세션
   - 두 세션이 독립적으로 캐시됨, 하나를 고쳐도 다른 쪽은 별도 확인 필요
